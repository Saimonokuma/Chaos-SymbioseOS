What is "tyd"?


    The CHAOS auto discovery daemon is called "tyd" - pronounced "tie-dee" (like "tidy"). It is an implementation of the TNP protocol, through a brutally butchered omdiscd source framework; it is a single process/single thread binary that operates as both client and server in the TNP process. Tyd can be exectuted from /sbin/tyd on a running CHAOS node.



How does it work?


    Tyd has two components, Terrence and Phillip. Terrence acts as a client, interrogating Phillip, the server. On the first node in a cluster, Tyd is executed without any paramaters, forcing it to run without Terrence; we'll come back to Phillip later. On every other node, Tyd is executed with one paramater; an IP address for another node. Tyd can now tell Terrence where one Phillip is. Terrence connects to Phillip and, once they have met and exchanged their formal greeting, Terrence begins to interrogate Phillip.

    Terrence is going to be busy for a little while. First, he retrieves the mosix map from the Phillip he was told about; he updates the local CHAOS node with this map. Then, he connects to every Phillip in the map (cluster), one by one, starting with the first (lowest numbered node-id) and moving through to the last. As he visits each Phillip, Terrence asks him to add the new CHAOS node to his own map. Phillip does this, and returning the current total number of nodes to Terrence. If the total number of nodes that a given Phillip knows about, is more than the total number of nodes Terrence is expecting, then Terrence has been passed by another Terrence; he aborts his cluster addition interrogation, and starts again - asking the first node for a new map.

    Phillip, relatively speaking, has a much easier job. Once Terrence has been successful in joining the cluster (map) he goes away, leaving a Phillip in place on the new CHAOS node. This Phillip is like all or any other; contains a map of the entire cluster, is ready to serve maps to a passing Terrence, and to add new Terrences as they occur.

What is "get-m"?


    The CHAOS autodiscovery daemon, tyd, needed a little help. As mentioned above, tyd needs to know the IP address of just one node in the cluster (any node), so that the Terrence component of tyd can download the cluster map, and find all of the other nodes in the cluster. As you will see below, to specify this "master" node, you pass tyd the "-m" paramater.

    This new daemon get-m, pronounced "get em" (like a Brittish "get them"), removes the need to manually key in an IP address for Terrence to retrieve a map from. Instead, get-m uses a multicast sonar to try to find any nodes that may already exist on this LAN segment. If it finds any, it records them in /var/run/get-m.info for tyd to find.



How about a usage example?


    Before we begin, CHAOS-1.0 comes with get-m (read above), largely removing the need to perform the following. If you're using get-m and you can't find another node in your cluster, then you will need to perform the "second node in the cluster" step on the first node to be started in the new mulitcast domain (this LAN or VLAN segment).

    The first node in the cluster runs the daemon without parameters (there is no cluster to join);

            [192.168.1.12]# tyd


    The second node in the cluster runs the daemon with the IP address of the first node;

            [192.168.1.109]# tyd --master 192.168.1.12


    The third node in the cluster runs the daemon like the second, though it now has the luxury of choice. It can either use the IP of the first node in the cluster, or alternatively, the IP of the second node in the cluster;

            [192.168.1.87]# tyd --master 192.168.1.12
            [192.168.1.87]# tyd --master 192.168.1.109


    And so on.




CHAOS has stateful packet filtering?


    Fortunately, yes. The Mosix/openMosix architecture is incredibly insecure; from vulnerabilities in the respective implementations, all the way back to their insecure design. Since CHAOS 0.5, tyd has supported packet filtering using the linux netfilter kernel-level stateful packet filtering code.

    From CHAOS 0.7, if cryptography is disabled, then the filters employed allow only the node's http daemon and ty-daemon to be visible to the network, beyond those nodes registered in the cluster. If cryptography is enabled, then the filters employed allow only the node's http daemon, IPSEC's ESP protocol, and IKE's service port visible to the network, beyond those nodes registered in the cluster. With cryptography enabled, all openMosix and tyd communications are required to route via the ipsec0 interface, only.



So CHAOS has crypto; like 3DES under IPSEC?


    From CHAOS 0.7, tyd has supported the layer-3 VPN standard - IPSEC - with a default encryption transform of 3DES (Triple DES). CHAOS uses pre-shared-keys (PSKs) to authenticate the tunnel.

    The default value for the PSK is stored in /etc/ipsec.secrets. It is long and complicated, but it would still be most prudent avoid using the default. Note that using the default will ensure that differing tyd versions will not successfully communicate within the same cluster. See also the following sections on compatibility for details on CHAOS nodes in non-CHAOS-homogeneous clusters.

    Nb; CHAOS applies the default FreeSWAN encrypt, tunnel, compress and pfs transforms in a fully meshed topology ("n-1" tunnels per node). All tyd and openMosix communications are encrypted and encapsulated in the IPSEC tunnels.



I can't (don't want to) run a homogeneous CHAOS openMosix cluster ...


    Fortunately, in many cases this will not cause concern. ClusterKnoppix has adopted tyd into its distribution, allowing harmony to reign between the two. There is a special HowTo document on building a secure cluster, based on ClusterKnoppix and CHAOS. And, since writing the HowTo, Quantian (a derrivative of ClusterKnoppix) has also adopted tyd.

    The compatible versions that contain the CHAOS security features, are; CHAOS 0.7, ClusterKnoppix 2003-11-19-EN-cl1 and Quantian 0.4.9.2. Note that the latest tyd was not made available to ClusterKnoppix or Quantian authors prior to the release of CHAOS-1.0; check the distribution web sites for current compatible versions.



I want to run a defferent type of heterogeneous openMosix cluster ...


    If the distribution(s) you wish to use are not tyd-enabled, then you will need to disable tyd on CHAOS.

    Some of the more valuable features of CHAOS will cause great pain when attempting to use CHAOS nodes in openMosix clusters that contain non-CHAOS distributions. There are two ways to ease your suffering; try running tyd and omdiscd simultaneously, or remove tyd comlpetely, running omdiscd only. From a freshly booted node, try one of the following ..

    Running both discovery daemons simultaneously;

            [192.168.1.87]# tyd --filter off --filter init
            [192.168.1.87]# tyd --filter off [--master ip]
            [192.168.1.87]# omdiscd


    Running the openMosix discovery daemon only;

            [192.168.1.87]# tyd --filter off --filter init
            [192.168.1.87]# omdiscd


    Please remember, that disabling tyd will remove the unicast functionality and network level encryption that you may have enjoyed previously. Note too, that the results of running both daemons simultaneously may yield undefined results (such as pieces of cluster map discovered by unicast nodes that are not discovered by multicast nodes and vice-versa). The best solution for compatbility is probably reverting back to omdiscd only.



Why did you create another auto discovery process?


    The TNP protocol was created to solve the shortcomings of the omdiscd. The omdiscd does an awesome job of dynamically configuring the kernel with a common cluster-wide openMosix map; providing your cluster is within a single VLAN or LAN segment. This segment boundary limitation is not intentional - the multicast traffic used to maintain the cluster configuration is only isolated due to the routing infrastructure surrounding a given segment. However, having spent three days trying to get mrouted to bridge VLANs on a proxy-arp'd class-b network, it became evident that writing a unicast autodiscovery daemon would be far more productive.



Does tyd have any known limitations?


    At this stage tyd, like omdiscd, does not support node removal. Hooks have been coded, but the openMosix kernel is not accepting the node removal request -- more investigation is required.

Why Terrence and Phillip?


    Internetworking protocols, such as the autodiscovery protocol documented here, must strike a harmonious chord both without, as in their network operating environment, and within, between and among their peers. We believe that Terrence and Phillip are the personification and embodiment of this ephemeral harmony - and are, or should be, role models for us all.
