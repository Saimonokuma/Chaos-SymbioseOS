What is a "certaintyd"?


    Glad you asked. This neat little daemon is a blessing for an operating environment (SSI) that can be spread across thousands of hosts. CHAOS, among its peers, is excessive in its approach to security; containing three distinct security improvements. One of those improvements, certaintyd, is an integrity checker that will reboot a node if notable exploitation occurs (the others being dynamic stateful packet filtering and layer-3 data encryption - in tyd).

    A word to the wise; certaintyd is still included in CHAOS-1.x but no longer runs from any boot menu option. This is due to the SSH public key sharing that is done in tyd, which cannot presently be disabled (this disable feature is awaiting a little protocol redevelopment in tyd itself). If you desperately want certaintyd enabled, you should comment out the SSH key sharing in tyd.



How does it work?


    The certaintyd takes an md5(a) checksum of the root disk (/dev/ram0) once every two seconds, 30 seconds after startup. Should the CHAOS running disk image change (even one iota) the certaintyd will send a SIGTERM to process one (initd). The upshot? If the CHAOS system gets hacked, then any change to the running image will result in the whole node being kindly rebooted.



It doesn't look fool-proof to me!


    The certaintyd process can be killed .. a hacker need not write to the root image to break into the system .. there are a number of "flaws" to the concept. However, the reassurance that comes from certaintyd will be that "someone is watching" node 106 at 3am while you're not.

    Nb; the process has not been sig-trapped, so that it can be killed in the event that software needs to be added to the node directly -- by the owner .. hopefully ;-)



If you have problems with nodes rebooting for no reason ..


    FYI; It has been observed that in some cases certaintyd will kill init for no apparent reason. It is still not fully understood why the root (memory) file-system would invalidate its checksum, in these cases. However, to date, this issue has only emerged when CHAOS is booted under VMware.

    As a second FYI, recent (1.0-pre) versions of Busybox seem to produce a single eroneous md5sum output, prior to a consistent one. While certaintyd has been updated to reflect this issue, there is still much concern as to why the issue exists at all.
