  1. HowTo :: Build a supercomputer with a business card from your wallet


        This document is a good read for those who have not yet constructed an openMosix cluster from the CHAOS distribution. What you will learn here, is how to construct a distributed processing environment from any of the network connected pentium-grade computers that you already have access too.

        It is beyond the scope of this document to convey details on burning CDROM images, using unix/linux shell commands, constructing IP networks, or helping you with your homework (that includes your thesis). This document will not go into detail regarding the individual CHAOS OS components, these details can already be found amongst the supplied documentation.



    Prerequisites


        You should be familiar with the general administration and operation of an openMosix cluster. You should also be familiar with concepts such as IP networking, routing, and address management. And, if you are so inclined, it might be a good idea to be a little familiar with linux packet filtering (netfilter). If you are not confident in these areas, then acquire some playtime, Redhat and openMosix RPMs.

        It is recommended that you plan your cluster before you set out to install it. This practice will help you to avoid complicated, time consuming and expensive surprises. Cute advice from an OS that professes to build "adhoc" clusters; if it makes you feel better, we recommend you do your planning on a napkin that came with your last asian take-out ;-)



    Let's start already, I just want to get it running ..


        Too right! Life is short and there is no time to waste on sentences like this one! To get your cluster running immediately, you will need to perform the following four major steps.

        1.   Some minor preparation work must come first;

            o   Download the CHAOS-1.0 ISO image from the CHAOS web site ("about" has download info)
            o   Burn the ISO image to CDROM media (preferrably something cool)
            o   Acquire i586 (or better) PCs; each with CDROM drive, ethernet NIC and a console (serial will do)


        2.   Then, for each PC (node) you can perform the following;

            o   Ensure that this PC is capable of booting from its CDROM drive (check the BIOS configuration)
            o   Boot the node with the CHAOS disc
            o   If your LAN has dynamic IP allocation;
                  -   Press Enter (optionally; select "n1" if this is the first node)
            o   If your LAN is statically allocated;
                  -   Press F5 for help (and removing the boot prompt timeout)
                  -   Enter the boot command shown, but populate the correct IP detail for this node
            o   If the node's hardware is supported then within 24 seconds the CDROM will eject the disc
            o   As soon as that disc is ejected, you can repeat the process on the next node using the same disc


        4.   If your nodes are on differing LANs, then to join the cluster from all bar the first node;

            o   At the command prompt of the new node, enter; tyd --master 192.168.1.12
                 where 192.168.1.12 is substituted with the IP address of the first node


        Without any hardware issues, your first cluster should have two nodes running within about 10-15 minutes of downloading the CHAOS ISO image.



    Erm, I think its working, now what do I do?


        If you've followed through the instructions in the previous section, then you should have a running cluster. I'm sure that you've already figured out that you now have Gigacycles of idle CPU power returning your blank, drooling, stare. There are two handy little applications built into CHAOS that will help demonstrate the power of the openMosix cluster that you have at your fingertips; An openMosix performance monitoring tool and a test application.

        mosmon - The openMosix performance monitoring tool "mosmon" is really useful. Take the time out to play with mosmon, and to experiment with its display options. Once you've started mosmon, you should see all of the nodes in your cluster listed horizontally, with their respective load graphed vertically. Every two seconds, each node will produce a spike of load as the certaintyd attempts to validate that node's integrity.

            o   At the command prompt of any node, enter; mosmon


        testapp - The test application "testapp" is a tiny little program with one job; he has 2^32 fingers and toes, and he's going to play "this little piggy" until he gets to the end. Don't be concerned. Despite processing at millions of piggy's per second, the game won't end before this time tomorrow, some years from now ... well ... on commercially available computer equipment anywho. As you launch testapp instances you will see a huge load spike on the testapp running node, before it manages to offload (migrate) testapp instances to free nodes.

            o   At the command prompt of any other node, enter; testapp
                 do this for at least one testapp per node in your cluster


        When you've finished your little experiment, you can exit mosmon by pressing 'q', and you can terminate the testapp instances from the node that they were launched from, using ps and kill (as per regular unix-land tools and practices).

  2. HowTo :: Effectively cracking passwords using JtR and SSI technology


        This document is a good read for those security practioners or ICT auditors who have access to an openMosix cluster built from any openMosix enabled distribution. What you will learn here, is how to construct a parallel password cracking environment from the common password cracker, "John the Ripper" and the SSI cluster that you have already constructed.

        It is beyond the scope of this document to convey details on using John the Ripper, burning CDROM images, using unix/linux shell commands, constructing IP networks, or helping you with your homework (that includes your thesis). This document will not go into detail regarding the installation of openMosix, as this detail can already be found amongst the community's published documentation.

        Note, too, that Cisilia is beyond the scope of this document. Cisilia has been included in CHAOS since CHAOS was CHAOS, but it no longer meets our requirements.



    Prerequisites


        You should be familiar with the general administration and operation of an openMosix cluster and techniques or practices for using John the Ripper (JtR). If you are not confident in these areas, then acquire some playtime, a couple of Redhat boxes, and the openMosix and JtR RPMs.



    Here's one that was prepared earlier


        For those without patience for semantics, there are two options for you. The first is "plJohn"; authored by Johnny Cache, plJohn is a perl wrapper designed to run JtR in parallel instances, specifically for openMosix clusters (see the credits page for a link). The second is "forkjohn" (Scottish); authored by the CHAOS team, forkjohn is the ANSI/C version of plJohn, and is included natively in the CHAOS distribution (since version 0.7). While other software packages exist to parallelise JtR, they all seem to alter JtR in one way or another - which greatly restricts the user's ability to work with the JtR version s/he chooses.



    Why JtR is a special case


        A true single system image operating environment would not need any magic to make John the Ripper operate in a massively parallel way. John the Ripper can thread instances of itself to take advantage of multiprocessor machines, which would work ideally on an SSI. Unfortunately, openMosix is not a true SSI. It has the ability to manage and migrate fork'd processes, but not threaded ones; threaded processes remain locked on their home node.

        Therefore, JtR needs to be assisted to take advantage of the opportunity to parallel its work load. What we need to do, is find a way to get seperate instances of JtR executed, so that they can migrate, but also to operate on the same batch, so that they reduce the over all time taken.



    The theory


        Fortunately, John the Ripper has some very handy features that can be exploited, to obtain the functionality that we require -- and without editing the code.

        JtR has two key processes that we are interested in; a password feeder and a password encoder/comparer. The password feeder simply takes a dictionary / word file, and dumps it out as a set of words (passwords) for input into the encooder/comparer process. Alternatively, an option can be specified to have the feeder use the supplied dictionary to generate potential passwords based on the words and a set of rules. Either way, the output is a set of plain text passwords that are to be tested. The password encoder/comparer takes the password to be attempted, encodes or encrypts it per the designated algorythm, and then compares the result to the passwords that are being cracked.

        Sneakily enough, in practice the output of the feeder process can be redirected into a FIFO pipe. Then, as many encoder/comparer processes as can be handled (probably 1 per processor in the cluster, for maximum effect) can be executed, each reading from the FIFO pipe, instead of the dictionary file. Passwords entered into the FIFO will be created faster than they can leave. Encoder/comparer processes will each obtain a new password from the FIFO as it requires one, removing the password from the FIFO (preventing duplicated effort) and attempting that password against each entry in the file being cracked.

        In this way, individual and migratable processes are created, while all processes still work collectively on the single task.



    How to make it go really really fast


        At the command-line then, the theory can be applied in three very basic steps (no JtR code was hurt in the creation of this mirage);

        1.   Create a FIFO pipe;

            o   mknod /tmp/pipey p


        2.   Start the password feeder process;

            o   Either, the dictionary only;
                    john -session=/tmp/session.rec -stdout -w=/tmp/dictionary.txt > /tmp/pipey &

            o   Or, the dictionary plus rules;
                    john -session=/tmp/session.rec -stdout -w=/tmp/dictionary.txt -rules > /tmp/pipey &


        3.   Start n encoder/comparer processes;

            o   john -session=/tmp/sess-$$.rec -stdin -format="md5" /tmp/shadow < /tmp/pipey &


        Note that openMosix's MFS takes care of the translation between foreign nodes and the local (home) node when it comes to accessing the references that we've made to /tmp (ie. you don't need NFS to pull-off this stunt).



    Where do I get dictionaries or word files?


        As you will quickly realise, JtR's strength comes from the dictionaries that it can access (which is fair, considering that it is a dictionary-based attack that you are attempting to perform).

        The authors of John the Ripper sell a comprehensive dictionary CDROM that can be ordered from the JtR web site (see the credits page for a link). None of the CHAOS team members have ever seen or used this CD, so we cannot vouch for it.

        Note that dictionaries and word files can be found all over the internet. The CHAOS team have assembled a collection of dictionaries from a hand-full of public web sites that offer them openly. As the average CHAOS consumer will not be using CHAOS to create password cracking clusters, this collection has been put together into its own CDROM; the "supplemental" disc.

        No effort has been made to clean these dictionaries - to remove duplicate entries or to consolidate the word files. No statistical work has been done on these dictionaries, to estimate their effectiveness against given populations. Additionally, some of them are compressed, and will need to be uncompressed before they can be used.



    Where's JtR and forkjohn in CHAOS-1.0?


        CHAOS-1.0 took a harder line on the slave/compute nodes, and removed all usable non-slave software. Keep checking the CHAOS web site for a small image that will be able to be added to a USB device, and plugged into a running CHAOS node, to add the missing security software to the cluster.



    No kitty, its a bad kitty!


        Remember, the difference between a professional and a hacker is permission.

  3. HowTo :: Massively scaling SETI@home with/without a cluster


        This document is a written for those who are interested in the Berkeley SET@home project, who may be looking for a very simple and/or temporary way to contribute to this community project. What you will learn here is how to construct a SET@home multi-workunit processing environment from the CHAOS distribution and the text-based setiathome binary that is available via the SETI@home project site.

        It is beyond the scope of this document to convey details on burning CDROM images, using unix/linux shell commands, constructing IP networks, or helping you with your homework (that includes your thesis). This document will not go into detail regarding the installation of openMosix, as this detail can already be found amongst the community's published documentation.



    Prerequisites


        You should be familiar with the general administration and operation of an openMosix cluster and vaguely familiar with the ins and outs of the SETI@home software. If you are not confident in these areas, then acquire some playtime, a couple of Redhat boxes, and the openMosix and SETI@home RPMs.



    The theory


        SETI@home is a single threaded binary-distributed application, that operates by obtaining a chunk of data (work unit) from a Berkeley server, processing it for a few hours, then sending a summary (result) back. By its own design, the SETI@home application "setiathome" is not able to be sped up by anything other than faster CPUs. Symmetric Multi-Processing (SMP) and Single System Image (SSI -- or the openMosix version of it) clustering techniques are not going to reduce the processing time required by any one setiathome process because it is not, natively, a parallel process.

        Instead, additional output (overall perceived performance) can be gained by running multiple setiathome instances, simultaneously. And, there re two ways to achieve this, with CHAOS.



    Launching SETI@home on standalone machines


        From CHAOS 1.0, the boot media is able to launch a dedicated SETI@home node configuration. This configuration automatically removes all unnecessary services and adds the appropriate packet filtering, specifically for the SETI@home service. To boot a new SETI@home host, you need to perform the following basic steps.

        1.   Some minor preparation work must come first;

            o   Download the latest CHAOS ISO image from the CHAOS web site ("about" has download info)
            o   Burn the ISO image to CDROM media (preferrably something cool)
            o   Acquire i586 (or better) PCs; each with CDROM drive, ethernet NIC and a console (serial will do)

        2.   Then, for each PC (host) you can perform the following;

            o   Ensure that this PC is capable of booting from its CDROM drive (check the BIOS configuration)
            o   Boot the host with the CHAOS disc
            o   If your LAN has dynamic IP allocation;
                  -   Enter the boot option "s1"
            o   If your LAN is statically allocated;
                  -   Press F5 for help (and removing the boot prompt timeout)
                  -   Enter the boot command shown, but populate the correct IP detail for this host
                  -   At the end of the boot command, replace "TYD=1" with "SETI=1"
            o   If the host's hardware is supported then within 19 seconds the CDROM will eject the disc
            o   As soon as that disc is ejected, you can repeat the process on another computer


        The SETI@home client should immediately begin. In fact, one instance of SETI@home will be launched, per CPU found on the host it was started from. The SETI@home output has been verbosed and then sanitised to give you an indication of where the host may be up to at any given point in time.

        An installation of this type has been trialled successfully on site. A     photo series has been published, demonstrating the success of this technique.



    Launchnig SETI@home on a cluster


        See the HowTo above, entitled Build a supercomputer with a business card from your wallet, and build yourself a homogeneous CHAOS 1.0 based cluster. Then simply perform the following on your head node.

        1.   Launch one SETI@home instance per CPU (eg; 30 nodes of 2 CPUs each = 60)

            o   /usr/bin/mseti 60


        The desired number of SETI@home instances should begin, and migrate as required. The SETI@home output has been verbosed and then sanitised to give you an indication of where the cluster may be up to at any given point in time.

        Be aware that you are running from RAM, and that your RAMFS is of fixed size (around 12Mbytes). Each instance of SETI@home will require about 500Kbytes of space.



    How many woo's has CHAOS contributed?


        The openMosix community maintains their own SETI@home group, within the Berkeley project itself (see the openMosix community site for details). CHAOS is distributed with the user "CHAOS" pre-configured. The CHAOS user is registered to the openMosix group, and those group statistics are mirrored online, at the CHAOS openMosix SETI@home stats page, along with copies of the certificates that the CHAOS user has attained.






  4. HowTo :: Deploy a supercomputer nightly with Novell ZENworks (ZFD4)


        This document is a written for those who have an existing investment in Novell Netware, and Novell's desktop management system - ZENworks (ZenForDesktops 4 or ZFD4). Anyone currently operating ZFD4 will be familiar with the concepts required for this process.

        It is beyond the scope of this document to convey details on burning CDROM images, using unix/linux shell commands, constructing IP networks, configuring or operating Novell servers, or helping you with your homework (that includes your thesis). This document will not go into detail regarding the installation of openMosix, as this detail can already be found amongst the community's published documentation.



    Prerequisites


        You should be familiar with the general administration and operation of Novell Netware, ZENworks (ZenForDesktops4), and an openMosix cluster, and vaguely familiar with the ins and outs of the PXE/Intel network boot process. If you are not confident in these areas, then acquire some playtime, Netware and a couple of PCs, and/or maybe RedHat with the openMosix RPMs.

        The ZENworks PXE boot file pxe-zfd4.cfg, mentioned later in this article, was first publically released in CHAOS-1.4.



    The theory


        Organisations large enough to run Novell Netware, particularly with the ZFD4 management tool-suite, are highly likely to have a large and under-utilised desktop PC environment that is available for clustering for up to 16 hours per day.

        What many LAN Administrators may not be aware of, is the ease and relative security with which a cluster can be deployed and removed each day, whilst leaving the PC workstation resource unharmed. Additionally, it is not immediately obvious as to how ZFD4 will aid in the dynamic deployment of a night time super-computer environment; but it is easier than you think.

        All of the software that you will require is included in either ZFD4 or is on the CHAOS cdrom.



    Teaching ZENworks For Desktops new tricks (woof)


        The following process will disable the normal ZFD4 menu that appears on the workstation at boot time, and replace it with the instant CHAOS loader. Look further down in this section to see how to revert the process. A later revision of this HowTo will document the automation of the conversion of ZFD4, such that it is automatically scheduled each day.

        If ZFD4 is installed and operating correctly, then on the Netware server's SYSTEM volume, there will be a TFTP directory;

           \\server\sys\TFTP\

        If this directory does not exist then stop now as ZFD4 is not installed or not installed correctly. The TFTP directory contains all of the files that ZFD4 will use to transact between the remote desktop computers and the local server.

        The first file ZFD4 serves to the remote rebooting desktop computer is \\server\sys\TFTP\DINIC.SYS   DINIC.SYS is the pre-menu boot file that is used to collect and convey workstation-specific information that ZFD4 then uses for Imaging.


        To configure ZFD4 for CHAOS net booting (Nb: all instructions are oriented for working in the directory \\server\sys\TFTP\), then;

        1.   Backup Novell's DINIC.SYS;

            o copy DINIC.SYS DINIC.SYS.novell


        2.   Install CHAOS' DINIC.SYS;

            o copy [cdrom]:\boot\pxelinux.0 DINIC.SYS.chaos
            o copy DINIC.SYS.chaos DINIC.SYS


        3.   Add CHAOS boot files;

            o mkdir pxelinux.cfg
            o copy [cdrom]:\boot\pxe-zfd4.cfg pxelinux.cfg\default
            o copy [cdrom]:\boot\*.txt pxelinux.cfg\


        4.   Add CHAOS Operating System files;

            o mkdir chaos
            o copy [cdrom]:\chaos\*.* chaos\


        If you have been successful, then the next net-booted workstation will display the CHAOS PXE splash screen, at the step usually reserved for the ZFD4 PC boot menu.

        To return the configuration of ZFD4 to its orginal operation (Nb: all instructions are oriented for working in the directory \\server\sys\TFTP\), then;

        1.   Restore Novell's DINIC.SYS;

            o copy DINIC.SYS.novell DINIC.SYS


        Note that the switching of DINIC.SYS (between CHAOS' and Novell's) will determine which function is performed by the net-booting workstations.



    Automating ZFD4 Network Boot Service Switching


        The Novell Toolbox software package (see Credits for URL) allows DOS/Unix command-line-type commands to be executed on a Netware server. Follow the documentation to install the Toolbox package, before proceding. Once the Toolbox is running (toolbox.nlm) it should be possible to execute commands, such as dir on the Netware console.

        In the following procedure, the Netware server will be configured to automatically manage the workstation boot process. Once completed, the ZFD4 boot environment will be alternated with the CHAOS boot environment each day, such that the normal ZENworks boot process will run between 7am and 7pm, while the CHAOS boot process will run between 7pm and 7am.

        To automate the Win-to-CHAOS and CHAOS-to-Win process perform the following, at the Netware console;

        1.   Create chaoson.ncf;

            o Run edit chaoson.ncf
            o Add cp sys:tftp\DINIC.SYS.chaos sys:tftp\DINIC.SYS
            o Escape and save on exit


        2.   Create chaosoff.ncf;

            o Run edit chaosoff.ncf
            o Add cp sys:tftp\DINIC.SYS.novell sys:tftp\DINIC.SYS
            o Escape and save on exit


        3.   Modify the crontab;

            o Run edit crontab
            o Add the line 0  19  *  *  *  chaoson.ncf
            o Add the line 0  7   *  *  *  chaosoff.ncf
            o Escape and save on exit


        Workstations that are configured to PXE/net-boot will pick up the current DINIC.SYS, and behave appropriately.



    Automating Win-to-CHAOS Booting


        The following procedure documents the process for scheduling an automated Windows workstation reboot, to get workstations to consult ZFD4 for the currently preferred boot process (CHAOS or Windows). Note that the Policy that is about to be created, will run ipconfig (an arbitrary short-lived process) and shutdown the host, once completed;

        1.   Create a Policy;

            o Open ConsoleOne (get coffee)
            o In the container where your existing policies are, create a new Policy Package
            o Select Workstation Package and click Next
            o Name the Policy Package Scheduled Reboot and click Next
            o Click Finish


        2.   Edit the Policy;

            o Double-click the policy Scheduled Reboot
            o Click Add to display the Add Policy dialog
            o Enter the policy name Reboot and click Ok
            o Click Properties
            o Click Add to display the Item Properties dialog
            o Enter the name ipconfig.exe and click Ok
            o Click Ok on the Properties of Scheduled Reboot:Reboot dialog
            o Click Edit to display the Edit Policy Package Schedule dialog
            o Select the reboot time as desired, each day at 7pm, for example
            o Uncheck the Randomly dispatch policy during time period option
            o Click Advanced Settings to display the Advanced Schedule dialog
            o Check the Reboot after completion option
            o Click Ok on the Advanced Schedule dialog
            o Click Ok on the Edit Policy Package Schedule dialog
            o Click the Associations tab
            o Click Add to display the Select Objects dialog
            o Select all of the Workstations that you would like to have rebooted
            o Click Ok on the Select Objects dialog
            o Click Ok on the Properties of Scheduled Reboot:Reboot dialog


        Workstations that are logged in will have their applications terminated and will be forced to reboot. Workstations waiting at the ctrl-alt-del login screen will reboot as scheduled.



    Automating CHAOS-to-Win Booting


        The following procedure documents the process for scheduling an automated CHAOS node reboot, to get workstations to consult ZFD4 for the currently preferred boot process (CHAOS or Windows). Note that a pre-requisite of this process, is the CHAOS boot option HTTPD=2 which enables the CHAOS web-based reboot facility (in the pxe-zfd4.cfg). Additionally, the two paramaters to the script are the start and end IP addresses for the range used by the CHAOS-booted workstations;

        1.   Create /usr/local/bin/chaos-reboot.sh;

            o Mount the CHAOS CDROM mount /dev/cdrom /mnt/cdrom
            o Copy the script cp /mnt/cdrom/web/howto/reboot.txt /usr/local/bin/chaos-reboot.sh
            o Unmount the CHAOS CDROM umount /mnt/cdrom


        2.   Modify the crontab;

            o Run crontab -e
            o Insert 0  7   *  *  *  /usr/local/bin/chaos-reboot.sh 192.168.1.100 192.168.1.130
            o Escape and :wq


        While the     reboot.txt script has been developed for a bash shelled unix host, it could be adapted for Windows batch script, though not for Novell.



    Can't CHAOS be better integrated into ZFD4?


        An early attempt was made to adapt CHAOS to work natively within the Novell ZFD (ZENworks For Desktops) agent, so that the labs could be booted automagically, using the Netware SOE management tools.

        However, Novell took the strange route of having ZENworks net-boot a workstation to DR-DOS, to use loadlin to boot to Linux, to perform the disk-level work required, in an early ZFD4. Unfortunately, the DR-DOS ramdisk was too small and limitations on the native loadlin were such that CHAOS was unable to be loaded as the result of a ZENworks menu selection.

        Dazed and confused by first contact with the young ZFD4, by the time patches for ZFD4 came around, the CHAOS team had already decided to treat ZFD as an expensive DHCP and TFTP server for the managed LAN. The resultant procedure can be found above.



    Has this been used in production anywhere before?


        Macquarie University has tested this deployment amongst one of its production ZFD4 SOE environments in one of the resident business colleges. This particular college runs approximately 180 computers in six labs of about 30 Compaq EVO workstations (1 x P4 @ 2.4Ghz), each running Windows 2000.
